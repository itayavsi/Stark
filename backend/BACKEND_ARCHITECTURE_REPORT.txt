4. API ARCHITECTURE
================================================================================

4.1 AUTHENTICATION FLOW
-----------------------

Current implementation uses simple base64 encoding (NOT SECURE):
  1. User logs in via POST /auth/login
  2. Server returns base64(username:role) as token
  3. Client sends token in Authorization header
  4. Server decodes and validates

SECURITY RISK: Base64 is not encryption. Token can be easily decoded.
RECOMMENDATION: Replace with JWT tokens (see Section 7)

4.3 KEY WORKFLOWS
------------------

A. Quest Creation Flow:
   1. POST /quests/ with quest data
   2. storage.save_quest() inserts into open_quests
   3. quest_geometries record created with status='missing'
   4. Response returned

B. Quest Completion Flow:
   1. User marks quest "Done" in UI
   2. POST /quests/{id}/status {status: "Done"}
   3. quest_service.complete_quest()
   4. move_quest() moves quest between open_quests and finished_quests
   5. Geometry moved to finished_quest_geometries with accuracy values

C. Geometry Upload Flow:
   1. User uploads shapefile via multipart form
   2. geometry_service.save_quest_polygon_geometry()
   3. File saved to storage/uploaded_shapes/{quest_id}/
   4. GeoPandas reads and reprojects to WGS84
   5. geometry_storage.upsert_quest_geometry() saves JSONB
   6. Frontend queries /geometry/catalog for display

6. SCALABILITY ISSUES & LIMITATIONS
================================================================================

6.1 DATABASE CONCERNS (HIGH PRIORITY)
-------------------------------------

Issue 1: NO CONNECTION POOLING
Impact: Neon serverless creates new connection per request
Cost: Connection overhead on every request
Solution: Use connection pooling (asyncpg + pool)

Issue 2: MISSING INDEXES ON FREQUENTLY QUERIED COLUMNS
Impact: Full table scans for status, assigned_user, date queries
Solution: Add indexes (see section 3.3)

Issue 3: UNION QUERIES IN GEOMETRY_CATALOG
Code location: storage.py:87-93
Impact: Runs on EVERY geometry catalog request
Problem:
  SELECT ... FROM open_quests UNION ALL SELECT ... FROM finished_quests
Solution: Separate queries, caching, or read replicas

Issue 4: NO PAGINATION
Code location: geometry_storage.py:get_ready_geometry_records()
Impact: Loads ALL records into memory
Solution: Add cursor-based pagination

Issue 5: NO QUERY CACHING
Impact: Repeated identical queries hit database every time
Solution: Add Redis caching layer


6.2 FILE STORAGE CONCERNS (HIGH PRIORITY)
-----------------------------------------

Issue 1: LOCAL FILESYSTEM
Impact: Not shared across multiple server instances
Problem: If you scale to multiple instances, files won't be accessible
Solution: Migrate to S3-compatible storage

Issue 2: NO FILE SIZE LIMITS
Impact: Large shapefile uploads could exhaust disk
Solution: Add upload limits in FastAPI endpoint

Issue 3: NO CLEANUP POLICY
Impact: Uploaded files are never deleted
Solution: Implement cleanup job for old geometries

Issue 4: BLOCKING FILE I/O
Code location: geometry_service.py:save_quest_polygon_geometry()
Impact: Blocks async event loop during file processing
Solution: Use async file operations or run in thread pool


6.3 APPLICATION CONCERNS (HIGH PRIORITY)
----------------------------------------

Issue 1: BLOCKING PGSQL IN ASYNC CONTEXT
Impact: psycopg is synchronous, blocks event loop
Solution: Replace with asyncpg

Issue 2: NO RATE LIMITING
Impact: DDoS vulnerability
Solution: Add Redis-based rate limiter (slowapi)

Issue 3: NO CACHING LAYER
Impact: Every request hits PostgreSQL
Solution: Add Redis caching

Issue 4: BASE64 "AUTH" (CRITICAL SECURITY)
Impact: No real security, easily decoded
Solution: Replace with JWT (see Section 7)

Issue 5: CORS ALLOW_ALL (SECURITY RISK)
Impact: Allows requests from any origin
Solution: Configure specific allowed origins


6.4 GIS PROCESSING BOTTLENECKS (MEDIUM PRIORITY)
-------------------------------------------------

Issue 1: LARGE SHAPEFILE MEMORY LOADING
Impact: GeoPandas loads entire file to memory
Solution: Stream processing, geometry simplification

Issue 2: NO GEOMETRY SIMPLIFICATION
Impact: High-res polygons cause large GeoJSON payloads
Solution: Simplify geometry for display zoom levels

Issue 3: SYNCHRONOUS CRS TRANSFORMATION
Impact: pyproj operations block
Solution: Run in thread pool or use async processing

Issue 4: NO GEOMETRY INDEXING
Impact: PostGIS spatial indexes not utilized
Solution: Migrate to PostGIS for spatial queries

7.3 AUTHENTICATION (CRITICAL)
------------------------------

Replace Base64 with JWT:

  ```python
  from jose import JWTError, jwt
  from passlib.context import CryptContext
  from datetime import datetime, timedelta

  SECRET_KEY = os.getenv("JWT_SECRET_KEY")
  ALGORITHM = "HS256"
  ACCESS_TOKEN_EXPIRE_MINUTES = 30

  pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

  def verify_password(plain_password: str, hashed_password: str) -> bool:
      return pwd_context.verify(plain_password, hashed_password)

  def create_access_token(data: dict):
      to_encode = data.copy()
      expire = datetime.utcnow() + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
      to_encode.update({"exp": expire})
      return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

  def decode_token(token: str):
      try:
          payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
          return payload
      except JWTError:
          return None
  ```


7.4 GIS PROCESSING OPTIMIZATION
-------------------------------

Geometry Simplification:

  ```python
  from shapely.simplify import simplify
  from shapely.geometry import shape, mapping

  def simplify_geometry(geojson: dict, tolerance: float = 0.0001) -> dict:
      """Reduce polygon complexity for display at different zoom levels"""
      geom = shape(geojson)
      simplified = simplify(geom, tolerance, preserve_topology=True)
      return mapping(simplified)

  # Different tolerances for different zoom levels:
  # z > 14: tolerance = 0.0001 (high detail)
  # z 10-14: tolerance = 0.001 (medium)
  # z < 10: tolerance = 0.01 (low detail)
  ```

14. CONCLUSION
================================================================================

The backend architecture demonstrates good software engineering practices with
clear separation of concerns. The current implementation is suitable for
small to medium workloads (up to ~50,000 quests).

For large-scale deployment (100,000+ quests), the following architectural
changes are required:

1. Async database drivers (asyncpg)
2. Redis caching layer
3. Object storage for files (S3)
4. JWT authentication (security)
5. Horizontal scaling infrastructure

The most impactful quick wins are:
- Adding database indexes (immediate improvement)
- Implementing Redis caching (10x performance gain)
- Replacing base64 auth with JWT (security)

These changes can be implemented incrementally without major refactoring
and will provide immediate benefits to the application.
